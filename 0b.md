# kotoba-gym Phase 0b 仕様書 (音声ループ検証)

## 0. 前提

Phase 0a(AI頭脳の検証)が完了し、テキストCLIで PM ペルソナとの会話・チェックリスト判定・フィードバック生成が成立していることが前提。本フェーズは **Phase 0a のコアロジック(`session.ts`)を再利用** する。

## 1. Phase 0b のゴール

**検証対象: 音声対話が「生きた会話」として成立するか**

- STT → LLM → TTS のトータルレイテンシが許容範囲か
- 沈黙で「あれ?」とならない会話テンポが出せるか
- ユーザーが話し終わってから相手の声が返るまで、自然に感じるか

体験が破綻するのはほぼ「待ち時間」が原因。**ここで詰まると見た目をどれだけ作っても無駄** なので、キャラUIを作る前に潰しきる。

## 2. 非ゴール(このフェーズではやらない)

- PNGtuber的キャラ表示(静止画1枚 or 文字のみで可)
- 会議風UI(デザインはほぼ未整備で良い)
- 理解度ゲージの可視化
- 選択肢モード / テキストモード(Phase 2で)
- 複数ペルソナ対応
- アプリ化(Web のみ)
- 本番デプロイ(ローカル or 軽く Vercel / Cloudflare 程度でOK)

## 3. 技術スタック

### フロントエンド

- **フレームワーク**: Vite + 素の TypeScript(React すら今は不要、軽く済ます)
  - ただし React/Next を使いたければ Next.js App Router でも可
- **STT**: **Web Speech API**(ブラウザ標準、無料、日本語対応、ストリーミング対応)
  - 互換性: Chrome / Edge で動作確認、Safari/Firefox は後回し
- **TTS再生**: Audio要素でストリーミング再生、または Web Audio API
- **通信**: Server-Sent Events (SSE) または WebSocket(SSEで十分、シンプル優先)

### バックエンド

- **フレームワーク**: **Hono**(軽量・エッジ対応・TypeScript ネイティブ)
  - Node.js 上で `@hono/node-server` で起動
- **LLM**: Gemini(Phase 0a と同じ、`session.ts` をそのまま利用)
- **TTS**: **Google Cloud Text-to-Speech**(日本語音声の品質とストリーミング対応)
  - 代替候補: OpenAI TTS, ElevenLabs(日本語は ElevenLabs が表現力高いが高コスト)
  - VOICEVOX(将来キャラ付け時に検討、ローカル実行 or 有料ホスティング)

### モノレポ構成検討

Phase 0a のリポジトリに `apps/web`, `apps/server`, `packages/core` の構成で pnpm workspaces を切る。`session.ts` を `packages/core` に移して共有する。

## 4. アーキテクチャ

```
┌─────────────┐                  ┌──────────────┐                 ┌────────────┐
│  Browser    │                  │  Hono Server │                 │  Gemini    │
│             │                  │              │                 │            │
│ Web Speech  │── 最終テキスト ─▶│  /session    │── Persona ────▶│            │
│  API (STT)  │                  │   /reply     │                 │            │
│             │                  │              │◀── stream text ─│            │
│             │◀── SSE text ─────│              │                 │            │
│             │                  │              │── Judge ──────▶│  (別コール)│
│             │                  │              │                 └────────────┘
│             │                  │              │                 ┌────────────┐
│             │◀── audio chunk ──│   /tts       │── text ───────▶│ Google TTS │
│  Audio      │   (streaming)    │              │◀── audio ──────│            │
│  playback   │                  │              │                 └────────────┘
└─────────────┘                  └──────────────┘
```

### 会話ループの1ターン

1. ユーザー発話開始 → Web Speech API が interim transcript を逐次表示
2. ユーザー発話終了(無音検知 or 明示ボタン)→ final transcript を `/reply` に POST
3. サーバーで:
   - a. Persona(PM返答生成)を Gemini にストリーミング要求
   - b. Judge(チェックリスト判定)を **並列・fire-and-forget** で発火(返答生成を待たせない)
4. Persona のテキストチャンクを SSE でフロントに流す
5. フロントは **文単位(句点で区切り)で TTS を呼び、音声を順次再生**
6. 最後の文まで再生 → ユーザー発話待ちに戻る

## 5. レイテンシ目標と計測

### 目標値

| 区間                                 | 目標  | 許容上限 |
| ------------------------------------ | ----- | -------- |
| ユーザー発話終了 → 1音節目が聞こえる | 1.5秒 | 3.0秒    |
| STT final transcript 確定            | 0.3秒 | 0.8秒    |
| LLM 初トークン到達 (TTFT)            | 0.8秒 | 1.5秒    |
| TTS 初chunk到達                      | 0.4秒 | 1.0秒    |

### 計測方法

セッション中に各区間の ms を計測し、終了時にコンソールに出す:

```ts
// 例: performance.now() で各区切りを記録
const marks = {
  userSpeechEnd: 0,
  sttFinal: 0,
  llmFirstToken: 0,
  ttsFirstChunk: 0,
  firstAudioPlay: 0,
};
```

**「数字で判定しようとしない」**。数字は目安。最終判断は「自分が会話して自然に感じるか」。

## 6. レイテンシ最適化テクニック

目標が出なかった時に試す手札(全部入れる必要はない、必要に応じて):

### a. 並列化

- Judge と Persona を並列発火(Persona の stream を止めない)
- Judge の結果はターン終了後にマージでOK

### b. ストリーミングTTS

- Gemini の出力を受け取りながら、**句点 or 読点で区切って即 TTS に流す**
- 最初の1文の TTS が返り始めたらすぐ再生開始
- 2文目以降は audio queue に積む

### c. フィラー挿入

- LLM 初トークン待ちが長い時、**先に「えーっと…」「なるほど」** を固定音声で鳴らす
- 本返答が来たら途切れないようクロスフェード、または次の句から繋ぐ
- 実装負荷と引き換えに大きな体験改善、**最終手段として検討**

### d. PM返答を短くする

- ペルソナプロンプトに「返答は2〜3文以内、質問は1つ」と明記
- 長文返しは TTS 時間も延びるので、**短さそのものがレイテンシ最適化** になる

### e. STT 側の無音検知チューニング

- Web Speech API の `onspeechend` タイミング
- 無音1秒で終了扱いにするなど(早すぎると途中で切られる、遅いと沈黙が長引く)

## 8. バックエンド構成

```
apps/server/
├── package.json
├── tsconfig.json
├── src/
│   ├── index.ts             # Hono サーバ起動
│   ├── routes/
│   │   ├── session.ts       # POST /session (新規作成), GET /session/:id
│   │   ├── reply.ts         # POST /reply (SSE) 返答生成+ストリーム
│   │   ├── judge.ts         # 内部用(必要なら公開)
│   │   └── tts.ts           # POST /tts テキスト→音声バイト(stream)
│   ├── sessions.ts          # インメモリのセッションストア
│   └── tts/
│       └── googleTTS.ts     # Google Cloud TTS クライアント
└── .env.example
```

### エンドポイント

- `POST /session` → セッション作成、`sessionId` 返却
- `POST /reply` (SSE) → body: `{ sessionId, userUtterance }` → `data: { type: "token", text }` を連続送信、`data: { type: "done", checklist, isEnded }` で終わる
- `POST /tts` → body: `{ text }` → audio/mpeg のストリーム返却

セッション状態は当面**メモリ保持**(プロセス再起動で消える、本フェーズは気にしない)。

## 10. 動作確認基準(このフェーズの完了定義)

以下を全て満たせば Phase 0b 完了:

- [ ] マイクボタンを押して話し、3分の会話が音声で最後まで通る
- [ ] ユーザー発話終了から相手の声が出始めるまで、体感で「会話している」と感じる
- [ ] 3回連続プレイして一度も致命的なバグ(無限ループ/無音/クラッシュ)が出ない
- [ ] Shoya本人が「これ音声で練習する意味あるな」と感じる

数値基準を設けているが、**最後の定性基準が最優先**。レイテンシ目標を達成していても「会話してる感」が出ないなら、プロンプトや間の取り方を見直す。

## 12. 将来拡張への配慮

- TTS プロバイダは抽象化しておく(Google → VOICEVOX → ElevenLabs 差し替え可能に)
- STT も同様(Web Speech API → Whisper API に切り替えられる構造に)
- SSE のイベントタイプは **後で拡張しやすい discriminated union** にする(例: `type: "emotion"` で感情イベントを追加予定、等)
- セッションはメモリ保持 → Phase 1 で Redis or SQLite に移行想定

## 13. 判断が必要な点(着手前に決めること)

- **マイクモード**: ホールド押下 / タップでトグル / 自動無音検知 のどれをデフォルトにするか
  - → 推奨: タップでトグル。自動検知は誤検知が体験を壊す
- **キャンセル機能**: ユーザーが話し始めたら相手の発話を止めるか
  - → Phase 0b では **「相手が話し終わるまで待つ」** シンプル版で可、将来の対応を塞がないようフラグを用意
- **音声再生中の字幕同期**: 文単位でハイライトするか、全文表示するか
  - → 全文表示で十分、同期は Phase 2 以降
