# kotoba-gym 修正タスクリスト

## 今回の問題意識
現在のお題詳細画面では、ユーザーがテーマに含まれる単語（例: API、キャッシュ）は理解できても、
「なぜそれを見直したのか」「どんな状況だったのか」といった前提を自分で補完しないと話せないケースがある。

その結果、以下の問題が起きている。

- 本来鍛えたい「話し方」より、即興で設定を作る力が必要になる
- ドメイン知識や想像力が強い人ほど有利になる
- フィードバックが「伝え方」以外のノイズを含みやすくなる
- ユーザーが話し始めるまでの負荷が高い

---

# 今回の修正方針
## 基本方針
**ユーザーに即興で世界設定を作らせない。**

お題詳細画面の時点で、話すのに必要な背景情報を最低限与えることで、
ユーザーが「何を話すか」を考える負荷を減らし、
「どう伝えるか」に集中できる状態を作る。

---

# 1. お題詳細画面の情報設計を修正する

## やること
現在の詳細画面にある以下の構成を見直す。

### 現状
- タイトル
- シチュエーション
- 相手の期待
- 今回の狙い
- 評価の5軸
- 回答を始める

### 修正後
- タイトル
- シチュエーション
- **背景 / 前提条件**
- 相手の期待
- 今回の狙い
- 評価の5軸
- 回答を始める

## 目的
ユーザーが「何が起きていたのか」を理解したうえで回答できるようにする。

## 背景 / 前提条件で入れる内容
1. 何が起きていたか
2. 何が問題だったか
3. 何を変えたか

## 量の目安
- 2〜3文
- 長くしすぎない
- 読めばすぐ話し始められる程度

---

# 2. お題文の設計ルールを見直す

## やること
今後のお題作成時に、以下のルールを追加する。

## ルール
### 説明型お題
- 背景事情を必ず与える
- ユーザーが状況を創作しなくても話せるようにする

### 即興型お題
- 前提をあえて曖昧にしてもよい
- ただし「即興で仮定を置く練習」であることが明確な場合のみ

## 目的
「説明型」と「即興型」が混ざって不自然になるのを防ぐ。

---

# 3. 既存お題を棚卸しする

## やること
今あるお題を確認し、以下の観点で分類する。

### 確認観点
- 背景情報が足りているか
- ユーザーが即興で設定補完しないと話せないか
- 本当に鍛えたいのは説明力か、それとも瞬発力か

### 分類
- A: そのままでよい
- B: 背景 / 前提条件を追加すべき
- C: 即興型お題として別扱いにすべき
- D: 問い自体が弱く、作り直した方がよい

## 目的
お題ごとの品質を揃える。

---

# 4. まず優先して直すべきお題を修正する

## 優先対象
「単語は分かるが、なぜそうなったのかが不明な説明系お題」

### 例
- APIキャッシュ戦略の説明
- DB分割の説明
- バッチ処理の見直し理由
- 非同期化を選んだ理由
- キャッシュ導入の説明
- リトライ戦略の説明

## 修正方針
タイトルは維持してもよいが、詳細画面に背景を追加する。

---

# 5. 「今回の狙い」の書き方も少し見直す

## やること
現在の「今回の狙い」を、より話の骨組みに寄せる。

### 現状例
- 最初に結論を置く
- 現状の問題と改善後の違いを分けて話す
- 数字や実例を1つ入れる

### 方向性
今の方向は良いので維持しつつ、
背景を追加したお題ではその背景に対応した骨組みになっているか確認する。

### 例
- 最初に見直した理由を一言で述べる
- 見直し前に起きていた問題を簡潔に話す
- どのように改善したかを説明する
- 改善結果や期待効果を1つ添える

## 目的
ユーザーが「何を順番に話せばよいか」を掴みやすくする。

---

# 6. UI上のラベルも必要なら見直す

## やること
「背景 / 前提条件」に使うラベルを検討する。

## 候補
- 背景
- 前提条件
- 状況の要点
- 今回の背景
- 起きていたこと

## おすすめ
- **背景**
または
- **状況の要点**

## 理由
硬すぎず、意味が分かりやすいから。

---

# 7. まず1問だけ実際に直して試す

## 対象
APIキャッシュ戦略の説明

## 修正版イメージ

### タイトル
APIキャッシュ戦略の説明

### シチュエーション
新しく入ったメンバーに、なぜ API レスポンスのキャッシュ戦略を見直したのか説明してください。

### 背景
最近アクセス数が増え、一部 API の平均応答時間が悪化していました。
特に商品一覧 API はピーク時に 900ms 前後まで遅くなり、画面表示の遅さも目立っていました。
そこでキャッシュ対象と TTL を見直しました。

### 相手の期待
相手はバックエンド経験が浅く、結論先出しで要点を知りたがっています。

### 今回の狙い
- 最初に結論を置く
- 見直し前の問題と改善内容を分けて話す
- 数字や具体例を1つ入れる

## 目的
まず1問で手応えを確かめる。

---

# 8. 修正後に確認すること

## 確認ポイント
- ユーザーが話し始めるまでの負荷は下がったか
- 「何を話せばいいか分からない」が減ったか
- フィードバックがより純粋に話し方評価になったか
- 回答内容のばらつきが減り、比較しやすくなったか
- それでも自由度は残っているか

---

# 9. 今回やることの優先順

## 優先度高
1. 詳細画面に「背景 / 前提条件」ブロックを追加
2. APIキャッシュ戦略の説明を修正
3. 他の説明型お題も棚卸しして、背景不足を洗い出す

## 優先度中
4. 説明型お題 / 即興型お題の分類ルールを決める
5. 「今回の狙い」を背景に合う形で少し調整する

## 優先度低
6. ラベル文言の細かい調整
7. 即興型お題の見せ方の整理

---

# 一言まとめ
今回の修正の本質は、

**「話し方の練習」に集中できるように、ユーザーに必要以上の設定補完をさせないこと。**

そのために、
- お題詳細画面に背景を追加する
- 説明型お題には前提情報を与える
- 即興型お題とは分けて扱う

この3つを進める。
