# Kotoba Gym 全MVP導線 新規実装計画

## Summary

`tasks/1.md` の新方針に合わせ、旧実装のドメインコードは移植せず、設定・構成だけ参考にして作り直す。

初回範囲は全MVP導線で、以下の流れを実装する。

```text
ホーム
↓
雑入力 / 文字起こし
↓
AI整理中
↓
材料整理
↓
結論選択
↓
伝える順番
↓
説明文生成
↓
リハーサル
↓
フィードバック
↓
保存 / 履歴 / 詳細
```

音声の文字起こしとAI生成は `apps/server` の Gemini API、履歴保存は Cloudflare D1 で行う。

## Key Changes

- `packages/core` を新設し、`DraftInput`, `OrganizedMaterials`, `ConclusionCandidate`, `SpeechPlan`, `GeneratedScript`, `RehearsalResult`, `Feedback`, `SessionRecord` などの共有 schema/type を `@kotoba-gym/core` から公開する。
- `apps/server` は Hono API として再構成し、音声文字起こし endpoint、AI生成 endpoint、D1-backed session endpoint を提供する。
- Gemini は `GEMINI_API_KEY` 必須、`GEMINI_MODEL` 任意で読み込む。未指定時の既定値は `gemini-3-flash-preview`。
- D1 にはセッション全体を保存する。`ownerKey` で端末単位に履歴を分離し、MVPではログインを入れない。
- `apps/mobile` は Expo Router の初期タブ構成を廃止し、Stack ベースの全MVP導線に作り直す。端末には永続データ本体を持たず、`ownerKey` と進行中画面状態だけを扱う。

## API / Storage

AI endpoint:

- `POST /v1/transcribe`
- `POST /v1/organize`
- `POST /v1/conclusions`
- `POST /v1/speech-plan`
- `POST /v1/script`
- `POST /v1/feedback`

Session endpoint:

- `POST /v1/sessions`
- `GET /v1/sessions?ownerKey=...`
- `GET /v1/sessions/:sessionId?ownerKey=...`
- `PUT /v1/sessions/:sessionId`

D1 table は `sessions` を中心にし、以下を保持する。

- `id`
- `owner_key`
- `title`
- `raw_input`
- `materials_json`
- `conclusions_json`
- `selected_conclusion_json`
- `speech_plan_json`
- `script_json`
- `rehearsal_json`
- `feedback_json`
- `created_at`
- `updated_at`

録音ファイル本体は初回保存対象外。リハーサルは録音完了状態と必要最小のメタデータのみ保存する。

## Mobile Implementation

- ホームではシーン選択を置かず、音声の雑入力、文字起こし確認、テキスト修正、最近の整理を同じ画面で扱う。
- 雑入力は音声入力を主導線にし、録音後に文字起こし結果を表示する。テキスト入力は文字起こし修正とフォールバックとして残す。
- AI整理中画面は各 Gemini API 実行中の共通待機画面にする。
- 材料整理、結論選択、伝える順番、説明文生成は API 結果を編集/選択しながら session に反映する。
- リハーサルでは `expo-audio` で録音開始/停止とタイマーを実装する。文字起こし・録音アップロードは初回範囲外。
- フィードバックでは点数を前面に出さず、良かった点、改善点、次の一言、Before/After を表示する。
- 履歴/詳細は D1 の session API から取得し、再練習導線を提供する。

## Server / Core Implementation

- Core の schema は Zod で定義し、mobile/server の両方で request/response validation に使う。
- Gemini prompt は「採点しない」「勝手に決めつけない」「仮の整理として出す」「日本語で短く具体的に返す」を共通 system prompt に含める。
- API は JSON schema response を要求し、失敗時は `error.code`, `error.message` の形式で返す。
- サーバー永続化は D1 の session API に集約し、AI endpoint は stateless に扱う。

## Test Plan

- `packages/core`: schema の正常/異常ケースを `describe.each` または `test.each` で検証する。
- `apps/server`: Gemini client と D1 repository を mock し、各 API の validation、保存、取得、失敗時エラーを Vitest で検証する。
- `apps/mobile`: pure helper を優先して、入力補助判定、API payload 生成、session state 更新、表示文言生成をパラメーターテストで検証する。
- 変更後に `pnpm test`, `pnpm lint`, `pnpm format`, `pnpm typecheck` を実行する。

## Assumptions

- MVPではログインなし。端末で生成した `ownerKey` を使い、D1上の履歴を端末単位で分ける。
- 旧 `_old` のドメイン/API/画面コードは移植しない。Gemini client、Wrangler/D1設定、Expo依存関係の選定だけ参考にする。
- 録音ファイル保存、R2連携、ユーザー同期は初回範囲外。
- UIは「採点アプリ」ではなく「頭の中を片付ける作業台」として、温かいが甘すぎない文言・落ち着いた紙/ノート系のビジュアル方針で統一する。
