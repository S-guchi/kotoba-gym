# Kotoba Gym 画面設計

## 1. 画面設計の前提

Kotoba Gymは「話す練習アプリ」ではなく、**話す前に頭の中を整理するアプリ**として設計する。

そのため、画面設計では以下を重視する。

- 最初から録音させない
- 最初から綺麗に話させない
- ユーザーが雑に出した内容をAIが整理する
- ユーザーは「選ぶ」「直す」「話す」だけで進められる
- 採点される怖さより、「整理されていく安心感」を優先する
- 録音は最後のリハーサルとして扱う

---

```text
オンボーディング
↓
ホーム
↓

↓

```

# 3. 画面一覧

## 3.1 オンボーディング画面

### 目的

アプリの価値を一瞬で伝える。

### 伝えること

* 話せないのではなく、まだ整理できていないだけ
* 雑に出してよい
* AIが一緒に整理する
* 最後に自分の言葉で話せるようにする

### 画面構成

```text
[ロゴ / Kotoba Gym]

話せないんじゃない。
まだ整理できていないだけ。

ぐちゃぐちゃな考えを、
AIと一緒に「伝わる説明」へ。

[はじめる]
```

### 補足スライド案

#### 1枚目

```text
話す前に、頭の中を整理しよう
```

#### 2枚目

```text
思いついた順に話すだけでOK
AIが材料を分解します
```

#### 3枚目

```text
結論・背景・相談事項を整理して、
自分の言葉で話せるようにします
```

### CTA

```text
[はじめる]
```

---

## 3.2 ホーム画面

### 目的

ユーザーが「今から整理したいこと」をすぐ始められるようにする。

### 基本方針

ホームは学習アプリっぽくしすぎない。
「今日の課題」や「スコア」よりも、**今困っている説明を整理する入口**にする。

### 画面構成

```text
おかえりなさい

今日は何を整理しますか？

[仕事の相談]
[会議で話すこと]
[面接回答]
[パートナーに伝えたいこと]
[自由に整理する]

最近の整理

- CI導入の相談
- 転職理由の説明
- Pikoの改善方針
```

### 主要コンポーネント

#### 整理タイプカード

| タイプ                   | 用途                         |
| ------------------------ | ---------------------------- |
| 仕事の相談               | 上司・同僚に相談する前の整理 |
| 会議で話すこと           | 会議で発言する内容の整理     |
| 面接回答                 | 転職・面接回答の整理         |
| パートナーに伝えたいこと | 家族・夫婦間の相談整理       |
| 自由に整理する           | 用途を限定しない整理         |

### 重要な文言

```text
うまく話す必要はありません。
まずは頭の中にあることを出すだけでOKです。
```

### CTA

```text
[整理をはじめる]
```

# 4. 雑入力画面

## 4.1 目的

ユーザーが未整理のまま考えを出せるようにする。

この画面で最も重要なのは、**綺麗に話さなくてよい空気**を作ること。

## 4.2 画面構成

```text
思いついた順でOKです

うまく話そうとしなくて大丈夫です。
頭に浮かんでいることを、そのまま出してください。

[音声で話す]
[テキストで書く]

困ったら、下の質問に答えてみてください。

- 何について話したいですか？
- 何に困っていますか？
- 誰に伝えたいですか？
- 相手に何をしてほしいですか？
```

## 4.3 音声入力モード

```text
思いついた順で話してください

[録音中アニメーション]

話すことに詰まっても大丈夫です。
途中で止まっても、そのまま整理できます。

[停止]
```

### 録音後

```text
録音できました

文字起こし結果:

「CIの話なんだけど、今手動でruffとpytestを回してて...」

[この内容で整理する]
[録り直す]
[テキストを修正する]
```

## 4.4 テキスト入力モード

```text
思いついたことを、そのまま書いてください

[テキストエリア]

例:
今度の会議でCI導入について相談したい。
今は手動でruffとpytestを回していて...
```

### CTA

```text
[整理する]
```

## 4.5 入力補助

入力が少なすぎる場合は、エラーではなく補助する。

### 悪い例

```text
入力が短すぎます
```

### 良い例

```text
もう少しだけ材料があると整理しやすいです。

たとえば、これに答えてみてください。

- 今どういう状況ですか？
- 何に困っていますか？
- 誰に伝えたいですか？
```

## 4.6 空送信対策

無言送信・空テキスト送信時は責めない。

```text
まだ材料が少なそうです。

まずは一言だけでも大丈夫です。
「何について整理したいか」だけ教えてください。
```

# 6. 材料整理画面

## 6.1 目的

ユーザーに「あ、自分が言いたかったことはこれだ」と感じてもらう。

この画面が新Kotoba Gymの中核。

## 6.2 画面構成

```text
あなたの話の材料

AIが、話の中身を分解しました。
違っていたら修正できます。

[現状]
今はruffとpytestを手動で実行している

[困りごと]
毎回PRに結果を貼るのが面倒

[背景]
モノレポ構成なのでCIの設計に迷っている

[不安]
チェック漏れが起きそう

[目的]
GitHub Actionsで自動化したい

[相談したいこと]
まず最低限のCIから入れてよいか確認したい

[まだ曖昧なこと]
- workflowをどこまで分けるか
- 通知をどうするか

[次へ]
```

## 6.3 編集仕様

各項目はタップで編集可能。

```text
[現状] 編集
```

編集後に再整理してもよい。

```text
[この内容で再整理]
```

## 6.4 項目が空の場合

空項目は「未入力」ではなく、軽く表示する。

```text
[背景]
まだはっきりしていません
```

```text
[相手に聞きたいこと]
ここはもう少し整理できそうです
```

## 6.5 重要なUX

ユーザーが「AIに勝手に決められた」と感じないようにする。

### 文言

```text
これは仮の整理です。
違っていたら直してください。
```

# 7. 結論選択画面

## 7.1 目的

「結局何を言いたいのか」をユーザーと一緒に決める。

整理が苦手な人に自由記述させるのではなく、AIが候補を出して選ばせる。

## 7.2 画面構成

```text
一番伝えたいことはどれですか？

A
CIを導入して、手動チェックをなくしたい

B
モノレポに合ったCI設計を相談したい

C
まず最低限のruff / pytest自動化から始めたい

[どれも違う]
[候補を作り直す]
```

## 7.3 選択後

```text
選んだ結論:

まず最低限のruff / pytest自動化から始めたい

[この結論で進める]
```

## 7.4 「どれも違う」時

```text
では、近いものを一言で教えてください

[テキスト入力]

例:
本当はCIより、レビュー前のチェックを楽にしたい
```

その後、再度候補生成する。

## 7.5 内部データ

```ts
type ConclusionCandidate = {
  id: string;
  label: "A" | "B" | "C";
  text: string;
  selected: boolean;
};
```

# 8. 伝える順番画面

## 8.1 目的

選んだ結論をもとに、相手に伝わる順番を作る。

PREP法を直接押し付けるのではなく、内容に合わせた順番にする。

## 8.2 画面構成

```text
おすすめの伝え方

この順番だと、相手が状況を理解しやすくなります。

1. 今の運用
ruffとpytestを手動で実行している

2. 困っていること
PRに貼る手間とチェック漏れがある

3. やりたいこと
GitHub Actionsで自動化したい

4. 迷っていること
モノレポでworkflow設計に迷っている

5. 相談したいこと
まず最低限のCIから入れてよいか確認したい

[この順番で進める]
[順番を入れ替える]
```

## 8.3 並び替え

ドラッグで順番変更できると良い。

MVPではなくても可。

## 8.4 なぜこの順番か

補助説明を短く表示する。

```text
先に現状を伝えることで、相談内容の背景が伝わりやすくなります。
```

## 8.5 注意

「PREP法」「SDS法」などの名前は前面に出しすぎない。
必要なら補足として出す。

```text
今回は、結論より先に現状を少し伝える方が自然です。
```

# 9. 説明文生成画面

## 9.1 目的

整理した材料を、実際に話せる形に変換する。

## 9.2 画面構成

```text
30秒で話すなら

今はruffとpytestを手動で実行して、その結果をPRに貼っています。
ただ、毎回手間がかかるのと、チェック漏れが起きる可能性があります。
なのでGitHub ActionsでCI化したいです。
ただ、モノレポなのでworkflowをどこまで分けるべきか迷っています。
まずはruffとpytestだけ自動化する形で進めてよいか相談したいです。

[自分の言葉で練習する]
[1分版を見る]
[Slack文面にする]
[修正する]
```

## 9.3 表示パターン

### 30秒版

短く話す用。

### 1分版

背景を少し厚めにした説明。

### Slack文面

テキスト送信用。

### 相談メモ

会議前に見返す用。

## 9.4 MVPで優先するもの

MVPでは以下だけでよい。

```text
30秒版
```

余裕があれば以下。

```text
1分版
Slack文面
```

## 9.5 重要な文言

```text
この文章を丸読みする必要はありません。
次の画面では、キーワードを見ながら自分の言葉で話してみましょう。
```

# 11. フィードバック画面

## 11.1 目的

発話の上手さではなく、説明構造をレビューする。

## 11.2 画面構成

```text
今回の整理度レビュー

良かった点

✓ 結論が早めに出ていました
✓ 困りごとと相談事項が分かれていました

改善するともっと伝わる点

・「なぜ今CI化したいのか」が少し弱いです
・最後の依頼をもう少し具体的にすると、相手が返答しやすくなります

次に意識すること

最後は、
「まずはruffとpytestだけ自動化する形で進めてよいか確認したいです」
と締めると伝わりやすいです。

[もう一度話す]
[整理内容を保存]
[Slack文面にする]
```

## 11.3 評価項目

MVPでは5項目。

```text
結論の明確さ
話す順番
背景の量
具体性
相談事項の明確さ
```

## 11.4 スコア表示について

初期は点数を前面に出しすぎない。

### 避ける

```text
あなたの点数は42点です
```

### 良い

```text
今回よかったところ
次に直すともっと伝わるところ
```

スコアを出す場合は、詳細画面に控えめに出す。

```text
整理度: 3 / 5
```

## 11.5 Before / After

可能なら、ユーザーに成長感を出す。

```text
最初の内容

CIの話なんだけど、手動でruffとかpytestやってて...

整理後

まずはruffとpytestだけGitHub Actionsで自動化してよいか相談したいです。
```

この比較はかなり強い。

# 29. 最重要画面

最も重要なのは、以下の2画面。

## 1. 材料整理画面

ユーザーが、

```text
あ、自分が言いたかったことはこれだ
```

と感じる画面。

## 2. 説明文生成画面

ユーザーが、

```text
これなら話せそう
```

と感じる画面。

この2つが弱いと、Kotoba Gymの価値が出ない。

逆に、この2つが強ければ、録音機能が未完成でも価値検証できる。

---

# 30. 最終方針

画面設計の中心は、以下の体験を作ること。

```text
話せと言われる前に、
話せる状態まで整理してくれる。
```

そのため、画面の順番は必ず以下にする。

```text
雑に出す
↓
整理される
↓
結論を選ぶ
↓
順番を作る
↓
話す
↓
振り返る
```

最初に録音させない。

最初に採点しない。

最初に型を押し付けない。

まず、ユーザーの頭の中にある未整理な材料を受け止め、見える形に分解する。

これが新Kotoba Gymの画面設計の基本方針である。
